# RL100 剩余实现问题审计（2026-03-13）

本文是在前两轮修复之后，对 `3D-Diffusion-Policy` 中 RL100 实现做的进一步补充审计。重点只记录“目前仍然存在”的问题，不重复已经修复的项。

论文参考：
- RL-100: Performant Robotic Manipulation with Real-World Reinforcement Learning
- arXiv: https://arxiv.org/abs/2510.14830

---

## 结论摘要

当前实现还剩 6 类值得继续处理的问题，其中 4 类会直接影响训练语义：

1. `resume offline_iter_N.ckpt` 时不会恢复扩展后的 replay buffer，恢复训练语义错误。
2. `MetaworldDataset.__getitem__()` 在 episode 尾部用 padded reward 计算 chunk reward，会重复累计 terminal reward。
3. `_relabel_demo_rewards()` 与 `merge_episodes()` 会静默抹掉 `val_ratio / max_train_episodes` 约束。
4. RL 阶段里 `Recon/VIB` 系数虽然会被调节，但当前 PPO/online RL 损失根本不消费这部分正则，系数修改实际无效。

另外还有 2 类次级问题：

5. `reward_type=dense` 时，对无 reward 的 demo 数据采用常数 `10.0` 重标，不是真正的 shaped reward。
6. consistency runtime / distillation 对 `cross_attention` 条件形状支持不完整；`train_imitation_learning(val_dataset=...)` 参数也未真正使用。

## P0: 恢复训练时没有恢复扩展后的数据集

涉及位置：
- `3D-Diffusion-Policy/diffusion_policy_3d/rl100_trainer.py:1411-1423`
- `3D-Diffusion-Policy/diffusion_policy_3d/rl100_trainer.py:1466-1522`
- `3D-Diffusion-Policy/train_rl100.py:149-153`
- `3D-Diffusion-Policy/diffusion_policy_3d/rl100_trainer.py:1289-1299`

问题：
- checkpoint 里只保存了 policy / critics / transition / optimizer / iteration。
- checkpoint 没有保存 replay buffer、merged episodes、train mask 或 dataset sampler 状态。
- 但恢复时会读取 `offline_rl_iteration`，并直接跳过前面已经完成的 offline iterations。
- 结果是：`offline_iter_N.ckpt` 恢复后，模型状态来自“扩展过的数据集”，但 `current_dataset` 仍然是重新从初始 zarr 加载出来的初始 demo 数据。

影响：
- 这不是单纯的“少了历史数据”，而是训练语义错误。
- 恢复后的 transition model / critics / policy 已经对应 `D_m`，但继续训练时喂进去的却是退回到 `D_0` 或其 reward relabel 版本的数据。
- OPE、IQL、后续 collection gate 都会建立在不一致的数据状态上。

建议：
- 要么 checkpoint 同时保存 merged replay buffer / collected episodes；
- 要么显式禁止从 `offline_iter_N.ckpt` 直接继续训练，只允许从 `after_il.ckpt` 重新开始 offline loop。

## P0: Episode 尾部 chunk reward 使用 padded reward，终止奖励会被重复累计

涉及位置：
- `3D-Diffusion-Policy/diffusion_policy_3d/dataset/metaworld_dataset.py:175-203`
- `3D-Diffusion-Policy/diffusion_policy_3d/common/sampler.py:110-141`

问题：
- `SequenceSampler.sample_sequence()` 在 episode 尾部会把最后一个 timestep 复制到 padding 区域。
- `MetaworldDataset.__getitem__()` 当前直接从 `sample['reward'][chunk_start:chunk_end]` 里做
  `R_chunk = Σ γ^j r_{t+j}`。
- 因为 reward 也被 padding 复制了，所以最后一个 terminal reward 会在 padding 区域被重复累计。

例子：
- sparse reward 下，真实最后一个 chunk 如果只剩 1 个有效 step，理论上 chunk reward 应该接近 `1.0`。
- 当前实现会把后面 padding 出来的多个 `1.0` 一起折扣求和，得到大于 `1.0` 的 chunk reward。

影响：
- critic target 被系统性抬高，尤其是在 episode 尾部 decision 上。
- `done.any()` 仍然会正确为真，但 reward 已经失真。
- 这个问题对 sparse reward 更明显，因为 terminal reward 本来就是唯一的正奖励。

建议：
- chunk reward / done 应该基于 replay buffer 中的真实区间计算，而不是基于已经 padding 完成的 `sample`。
- 具体做法可以和 `decision_start_idx` 一样，先反推出真实 decision 起点，再在真实 episode 边界内截取 reward/done。

## P1: `val_ratio / max_train_episodes` 在 RL 开始后会被静默抹掉

涉及位置：
- `3D-Diffusion-Policy/diffusion_policy_3d/rl100_trainer.py:616-668`
- `3D-Diffusion-Policy/diffusion_policy_3d/dataset/metaworld_dataset.py:299-309`
- `3D-Diffusion-Policy/diffusion_policy_3d/config/task/metaworld_*.yaml`

问题：
- 初始化 dataset 时，task config 里的 `val_ratio` 和 `max_train_episodes` 会正常生效。
- 但一旦进入 RL pipeline：
  - `_relabel_demo_rewards()` 会把 `episode_mask` 直接改成全 `True`
  - `merge_episodes()` 也会把 `episode_mask` 直接改成全 `True`
- 也就是说：
  - 原本保留的 validation episodes 被重新并回训练集
  - `max_train_episodes=90` 这样的限制会在 RL 开始时失效

影响：
- 实际 offline RL 用的数据量和 task YAML 表达的不一致。
- 如果你以为 RL 阶段仍然只用 train split，那当前不是。
- `get_validation_dataset()` 之后也基本拿不到真正 hold-out 的验证集。

备注：
- 如果这是你有意为之，至少应该显式打印日志，不应静默覆盖 task 配置。

## P1: RL 阶段里 `Recon/VIB` 系数调节当前是“死配置”

涉及位置：
- `3D-Diffusion-Policy/diffusion_policy_3d/rl100_trainer.py:1268-1284`
- `3D-Diffusion-Policy/diffusion_policy_3d/rl100_trainer.py:1320-1325`
- `3D-Diffusion-Policy/diffusion_policy_3d/rl100_trainer.py:1390-1395`
- `3D-Diffusion-Policy/diffusion_policy_3d/policy/dp3.py:444-449`

论文对应：
- 论文在表示学习部分明确写了 `L_IL = L_DP + L_recon + L_VIB`，并讨论了 RL fine-tuning 时降低 `Recon/VIB` 系数的做法。
- 这意味着如果代码里保留了 “RL 时把 beta 降 10 倍” 的逻辑，那么 RL 损失里必须实际消费到这两项正则，系数修改才有意义。

问题：
- 目前 `total_reg_loss` 只在 BC/IL 的 `DP3.compute_loss()` 里被加入总损失。
- offline PPO 和 online PPO 只优化：
  - PPO loss
  - optional consistency distillation loss
- 两条 RL 路径都没有把 `obs_encoder` 的 `Recon/VIB` 正则加回去。

结果：
- `_apply_vib_betas(0.1)` / `_apply_vib_betas(1.0)` 会改 beta 数值；
- 但 RL 阶段这些 beta 不参与 loss 计算，所以实际上不起作用。

影响：
- 如果你的目标是对齐论文里“RL fine-tuning 期间仍保留但减弱 Recon/VIB 正则”的实现，这里当前不符。
- 这也是为什么现在日志里会显示 beta 被切换，但 PPO 行为并不会随之变化。

## P2: `reward_type=dense` 时 demo 重标不是真正 dense reward

涉及位置：
- `3D-Diffusion-Policy/diffusion_policy_3d/rl100_trainer.py:620-645`

问题：
- 对于没有 `reward/done` 的 demo zarr，当前 sparse relabel 是“最后一步 reward=1”，这个逻辑至少是自洽的。
- 但如果设置 `reward_type=dense`，代码会把每个 timestep 都标成常数 `10.0`。

影响：
- 这不是 MetaWorld 环境的真实 shaped reward。
- 只能算一个占位近似，不能叫 dense reward 重建。
- 如果未来你切到 dense 配置，critic 学到的是“伪 dense reward”。

备注：
- 你当前配置若一直用 sparse，这项不会触发。

## P2: consistency 路径对 `cross_attention` 条件形状支持不完整

涉及位置：
- `3D-Diffusion-Policy/diffusion_policy_3d/policy/consistency_policy.py:32-45`
- `3D-Diffusion-Policy/diffusion_policy_3d/model/rl/consistency_model.py:203-229`
- 对比参考：`3D-Diffusion-Policy/diffusion_policy_3d/policy/rl100_policy.py:123-139`

问题：
- `RL100Policy.encode_obs_global_cond()` 已经正确区分了：
  - `film` 等模式使用 `[B, -1]`
  - `cross_attention` 使用 `[B, n_obs_steps, -1]`
- 但 consistency runtime wrapper 和 distillation 在自己编码 obs 时，都无条件 reshape 成 `[B, -1]`。

影响：
- 当前默认 `condition_type=film` 时不会出问题。
- 一旦切到 `cross_attention`，consistency eval / distillation 的条件形状就和 teacher policy 不一致。

## P3: `train_imitation_learning(val_dataset=...)` 参数没有真正使用

涉及位置：
- `3D-Diffusion-Policy/diffusion_policy_3d/rl100_trainer.py:314-410`

问题：
- 函数签名里有 `val_dataset`。
- 但函数内部没有建立 validation dataloader，也没有计算 validation loss 或 early stopping。

影响：
- 这是一个工程完整性问题，不一定直接影响当前 RL100 主流程。
- 但如果你以为 IL retrain 带有独立 val 监控，当前并没有。

---

## 建议修复优先级

建议优先顺序：

1. 修 `resume` 数据集恢复问题。
2. 修 `MetaworldDataset.__getitem__()` 尾部 padded reward 计算。
3. 明确保留还是取消 `val_ratio / max_train_episodes`，不要静默覆盖。
4. 决定 RL 阶段是否真的要保留 `Recon/VIB` 正则；如果要，就把它接进 PPO 总损失。
5. 如果未来要支持 dense / cross_attention，再修剩余两项。
