## 1. Chunk-action 决策对象已统一

### 问题

旧实现里 reward / discount 是按 chunk 算的，但 critic / transition / OPE 用的是 chunk 的第一个 action，导致优化目标和论文的 chunk MDP 不一致。

### 修改

- 在 `diffusion_policy_3d/policy/dp3.py` 新增了统一的 chunk helper：
  - `action_chunk_start`
  - `action_chunk_end`
  - `chunk_action_dim`
  - `extract_action_chunk()`
  - `flatten_action_chunk()`
  - `predict_action_from_global_cond()`
- 在 `diffusion_policy_3d/dataset/metaworld_dataset.py` 修正了 chunk 对齐逻辑：
  - `reward` 改为对真正执行的 chunk 窗口求和
  - `done` 改为该 chunk 窗口内任一步终止即为真
  - `next_obs` 改为“下一次 chunk 决策时刻”的观测窗口，而不再是错位的单步后继
- 在 `diffusion_policy_3d/model/rl/transition_model.py` 中：
  - 输入 action 改为整段 chunk flatten 后的向量
  - `next_obs_feat` 改为来自 dataset 提供的 chunk-level `next_obs`
- 在 `diffusion_policy_3d/rl100_trainer.py` 中：
  - critics 初始化的 `action_dim` 改为 `chunk_action_dim`
  - `train_iql_critics()`、`offline_rl_optimize()`、`_evaluate_policy_amq()` 全部改为使用 chunk-level normalized action

### 结果

现在 chunk mode 下的：

- `Q(s, a)`
- `A(s, a) = Q - V`
- transition model
- OPE gate

已经对齐到“整段 action chunk 是一个决策”的论文定义。

## 2. OPE gate 改成了真实 policy-conditioned 多步 rollout

### 问题

旧实现里 AM-Q 在第 2 步以后退化成随机动作近似，不再是论文 Eq.(20) 的 policy-conditioned rollout。

### 修改

- 在 `dp3.py` 增加 `predict_action_from_global_cond()`，允许策略直接从编码后的 feature 条件采样 action
- 在 `rl100_trainer.py::_evaluate_policy_amq()` 中：
  - 每一步 rollout 都直接从当前 `cur_features` 调用 policy
  - 不再在 `h > 0` 时退化为随机动作
  - 使用修正后的 chunk-level normalized action 与 transition model 相互作用

### 结果

AM-Q 现在是真正的：

- policy-conditioned
- 多步
- feature-space

rollout，而不是第一步 policy、后面随机动作的混合近似。

## 3. Online RL 已从“回退到 offline objective”改成真正的 GAE + on-policy PPO

### 问题

旧实现虽然算了 GAE，但 policy update 最终还是调用 `offline_rl_optimize()`，本质上重新回到了 `Q-V` 的 offline objective。

### 修改

- 在 `diffusion_policy_3d/policy/rl100_policy.py` 中新增：
  - `sample_for_ppo_from_global_cond()`
  - `predict_action_with_trajectory()`

  作用是：在线 rollout 时直接保存 behavior policy 的 denoising trajectory 和 old log-prob。

- 在 `diffusion_policy_3d/env_runner/metaworld_runner.py` 中：
  - `run_and_collect()` 现在除逐步 episode 数据外，还会额外返回 `decision_steps`
  - 每个 `decision_step` 包含：
    - 当前 obs window
    - action chunk
    - chunk reward
    - done
    - next obs window
    - behavior trajectory
    - old log-prob

- 在 `diffusion_policy_3d/rl100_trainer.py` 中新增：
  - `online_rl_optimize()`

  这个函数会执行：

  - 从 `decision_steps` 构造 online rollout buffer
  - 使用 chunk-level reward 和 `gamma_chunk` 计算 GAE
  - 用 Eq.(21) 的 value regression 更新 `V`
  - 用 rollout 时保存的 old trajectory / old log-prob 做 on-policy PPO
  - 同步加入 consistency distillation loss

- `run_pipeline()` 的 Phase 3 不再调用 `offline_rl_optimize()`，而是改成调用 `online_rl_optimize()`

### 结果

现在 online RL 的数据流已经变成：

- fresh rollout
- GAE
- value update
- on-policy PPO

不再回退到 offline RL 的 `Q-V` advantage 目标。

## 4. 最后一个 deterministic DDIM step 已从 PPO likelihood ratio 中剔除

### 问题

旧实现把 final DDIM step 裁成了 `sigma_min^2` 的伪随机步，仍参与 Gaussian log-prob 和 PPO ratio，这不符合论文对“只对严格正方差步骤使用 likelihood objective”的要求。

### 修改

- 在 `diffusion_policy_3d/policy/rl100_policy.py` 中：
  - `get_variance_at_timestep()` 对 final step 直接返回 `0`
  - `denoising_step_with_log_prob()` 对 final deterministic step 返回零 log-prob，占位但不再解释为 Gaussian transition
  - `compute_ppo_loss()` 显式跳过 final deterministic step，不再把它纳入 PPO ratio

### 结果

PPO 现在只对真正的 stochastic denoising steps 求 likelihood ratio，final deterministic step 不再被误用。

## 5. Consistency model 已接入 rollout / eval / deploy 主路径

### 问题

旧实现里 consistency model 只训练，不参与 env runner / eval 脚本 / 最终部署路径。

### 修改

- 新增 `diffusion_policy_3d/policy/consistency_policy.py`
  - `ConsistencyPolicyWrapper` 把 1-step consistency model 包装成标准 `BasePolicy`
  - 这样 env runner 不需要改接口就能直接跑 CM

- 在 `rl100_trainer.py` 新增：
  - `get_runtime_policy(mode='ddim'|'cm')`

- 在 `collect_new_data()` 中接入：
  - `runtime.collection_policy`
  - 离线数据扩展时可以选择用 `ddim` 或 `cm` rollout
  - online RL 仍强制用 `ddim` 收集，因为 PPO 需要 behavior trajectory / old log-prob

- 在 `train_rl100.py` 中：
  - final evaluation 改为支持多个 runtime policy
  - 默认同时评估 `ddim` 和 `cm`

- 在 `eval_rl100.py` 中：
  - 支持 `runtime.eval_policy_mode=cm`
  - 会自动构造 `ConsistencyPolicyWrapper`

### 结果

现在 consistency model 不再只是训练时的旁路分支，而是已经能走：

- rollout
- final evaluation
- checkpoint evaluation

这三条主路径。

## 6. OPE reject 时已同步回滚 consistency model 和 optimizer 状态

### 问题

旧实现里 OPE reject 只回滚 diffusion policy / EMA policy，consistency model 会停留在被拒绝 teacher 的蒸馏结果上。

### 修改

- `offline_rl_optimize()` 现在在 PPO 前会额外 snapshot：
  - `consistency_model`
  - `policy_optimizer`
  - `consistency_optimizer`
  - `ema` helper 内部状态

- OPE reject 时会一起恢复这些状态

### 结果

被 reject 的 update 不会再留下：

- teacher / student 失配
- optimizer moment 污染
- EMA step 漂移

## 8. Checkpoint 加载做了兼容增强

### 问题

这次修复把 critics / transition model 的 action 维度从单步 action 改成了 chunk action。旧 checkpoint 如果直接恢复，会因为 shape mismatch 失败。

### 修改

- 在 `rl100_trainer.py` 中新增 `_load_matching_state_dict()`
- 对以下模块改成“尽量加载可匹配权重，不匹配则告警并跳过”：
  - `critics`
  - `consistency_model`
  - `transition_model`
- optimizer state 改为 `try/except` 恢复
- checkpoint 中额外保存 / 恢复 `ema_helper`

### 结果

旧 checkpoint 在面对本次结构调整时，不会再因为局部 shape mismatch 直接中断整个恢复流程。  
能复用的权重会继续复用，无法复用的部分会明确告警并保留当前初始化。

## 9. 修复了 consistency distillation 的 action 维度串线崩溃

### 问题

在前面把 critics / transition model 切换到 chunk-level action 之后，`RL100Trainer.__init__()` 里一度把这个 `chunk_action_dim` 也传给了 `ConsistencyModel(input_dim=...)`。

但 consistency model 的 student UNet 学的是完整 diffusion action trajectory，其每个时间步的 channel 数应当仍然是单步 `action_dim`，不能替换成 flatten 后的 chunk 维度。

这会导致训练时出现类似下面的卷积通道不匹配错误：

- `weight of size [512, 32, 5]`
- `expected input[256, 4, 16] to have 32 channels, but got 4 channels instead`

### 修改

在 `diffusion_policy_3d/rl100_trainer.py` 中显式拆分了两套 action 维度：

- `critic_action_dim = self.policy.chunk_action_dim`
- `model_action_dim = self.policy.action_dim`

并分别接到不同模块：

- `IQLCritics(..., action_dim=critic_action_dim)`
- `TransitionModel(..., action_dim=critic_action_dim)`
- `ConsistencyModel(..., input_dim=model_action_dim)`

### 结果

现在：

- critic / transition 继续工作在 chunk MDP 上
- consistency model 继续工作在原始 diffusion trajectory 的单步 action channel 上

两条路径的语义已经分开，不会再因为 chunk flatten 维度误传而在 consistency distillation 阶段崩溃。

## 10. 采集日志已区分环境原始回报和训练用 RL 奖励

### 问题

原先 `collect_new_data()` 打印的 `Reward` 实际上是 MetaWorld 环境返回的原始 shaped reward 累加，不是写回 replay buffer、也不是 critic / PPO 真正使用的 RL 奖励标签。

因此在 `reward_type=sparse` 时，也可能看到很大的 `Reward` 数值，例如 `1000+`。这会让训练日志看起来像是“全程在用 dense reward”，但实际上不一定如此。

### 修改

- 在 `diffusion_policy_3d/env_runner/metaworld_runner.py` 中：
  - 保留环境原始回报统计为 `mean_env_rewards`
  - 新增训练实际使用的标签奖励统计为 `mean_rl_rewards`
  - `reward_type=sparse` 时，`ep_reward` 仍然只在成功 episode 的最后一步赋值为 `1`
- 在 `diffusion_policy_3d/rl100_trainer.py` 中：
  - 采集日志改为同时打印 `EnvReturn` 和 `RLReward`
  - WandB 同时记录：
    - `collection/env_return`
    - `collection/rl_reward`
    - `collection/reward` 现在对齐到训练真正使用的 `RLReward`

### 结果

现在看到：

- `EnvReturn` 大，是环境 shaped reward 的表现
- `RLReward` 小，才是 sparse / dense 配置真正作用到 RL 优化的奖励

也就是说，`success=0.72` 且 `EnvReturn=1000+` 仍然可能是完全正常的 sparse reward 训练。

## 11. 修复了 merge 新 episode 后的 `episode_starts` 越界错误

### 问题

`MetaworldDataset.merge_episodes()` 在把新 rollout 追加进 replay buffer 后，虽然重建了 `SequenceSampler`，但没有同步刷新：

- `self.episode_ends`
- `self.episode_starts`

于是 dataset 继续用旧的 episode 边界缓存来解析新采样到的 index。  
当采样命中新加入的 episode 时，`np.searchsorted()` 会返回新的 episode id，但 `self.episode_starts` 仍停留在旧长度，最终触发：

- `IndexError: index 100 is out of bounds for axis 0 with size 100`

### 修改

- 在 `diffusion_policy_3d/dataset/metaworld_dataset.py` 中新增：
  - `_refresh_episode_boundaries()`
- 在以下位置接入刷新：
  - dataset 初始化时
  - `merge_episodes()` 完成追加后
- 同时在 `__getitem__()` 中对 `episode_idx` 增加了边界 clamp，避免极端边界条件再越界

### 结果

新 episode 被 merge 之后，dataset 的：

- replay buffer
- sampler
- episode 边界缓存

三者现在会保持一致，不会再在后续 IL / critic dataloader 中因为旧缓存访问新 episode 而崩溃。

## 使用方式

### 训练

默认现在会：

- 跑 offline RL
- 跑 online RL
- 最终同时评估 `ddim` 和 `cm`

命令示例：

```bash
python 3D-Diffusion-Policy/train_rl100.py task=metaworld_dial-turn
```

### 选择 offline 数据扩展时的 rollout policy

```bash
python 3D-Diffusion-Policy/train_rl100.py task=metaworld_dial-turn runtime.collection_policy=cm
```

说明：

- 这只影响离线数据扩展阶段的 rollout
- online RL 阶段仍会固定使用 `ddim`

### 单独评估 consistency policy

```bash
python 3D-Diffusion-Policy/eval_rl100.py \
  task=metaworld_dial-turn \
  checkpoint_path=/path/to/final.ckpt \
  runtime.eval_policy_mode=cm
```
