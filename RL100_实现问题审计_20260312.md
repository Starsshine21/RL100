# RL100 实现问题审计报告

审计日期：2026-03-12  
审计对象：`/home/yrz/RL-100/3D-Diffusion-Policy` 当前工作树  
论文基准：`RL-100: Performant Robotic Manipulation with Real-World Reinforcement Learning`（arXiv:2510.14830）

## 结论

当前仓库已经具备 RL100 的主要骨架：三阶段训练入口、IQL critics、PPO 式 diffusion actor、transition model、consistency distillation、OPE gate 都有实现。

但如果目标是“按论文复现 RL100 的训练逻辑和结论”，当前实现仍有几处关键偏差，其中 4 个属于硬问题：

1. `chunk-action` MDP 没有真的按“整段 action chunk 作为一个决策”实现。
2. OPE gate 里的 AM-Q 不是论文定义的 AM-Q，多步 rollout 在第 2 步后退化成随机动作近似。
3. Online RL 阶段没有按 Eq.(21) 做 on-policy PPO + GAE，最终仍回落到 offline objective。
4. PPO 对最后一个 deterministic DDIM step 仍计算 Gaussian log-prob，这和论文对“仅正方差步骤可用于 likelihood objective”的约束不一致。

此外还有 2 个中等级别问题：

5. consistency model 训练了，但没有接入 eval / rollout / deployment 主路径；OPE reject 时也不会回滚 consistency model。
6. 默认配置和论文完整 recipe 不一致，且默认恢复路径指向仓库外部，导致“开箱即跑”和“论文对齐复现”都不成立。

下面按严重度展开。

---

## P0: Chunk-action MDP 实际上被降成了“只看 chunk 的第一个 action”

### 论文要求

论文在方法部分明确区分了 `single-step` 和 `action-chunk` 两种控制模式。  
在 `action-chunk` 模式下，一个 chunk 应作为一个单独决策，chunk reward 用 `sum_j gamma^j r_{t+j}`，chunk 间折扣用 `gamma^n_c`。

### 当前代码证据

- 任务配置的动作维度仍是单步动作维度，而不是 chunk 决策维度：
  - `diffusion_policy_3d/config/task/metaworld_dial-turn.yaml:13-14`
- IQL critics 训练时只取 `batch['action'][:, 0, :]`：
  - `diffusion_policy_3d/rl100_trainer.py:452-471`
- Offline PPO 计算 advantage 时也只取 `batch['action'][:, 0, :]`：
  - `diffusion_policy_3d/rl100_trainer.py:796-815`
- Transition model 训练输入同样只用 chunk 的第一个 action：
  - `diffusion_policy_3d/model/rl/transition_model.py:297-326`
- OPE 在 `h=0` 时也只取 `action_pred['action'][:, 0, :]`：
  - `diffusion_policy_3d/rl100_trainer.py:695-699`

### 问题本质

代码里 reward 和 discount 已经按 chunk 处理了：

- `diffusion_policy_3d/config/rl100.yaml:82-89`
- `diffusion_policy_3d/dataset/metaworld_dataset.py:126-164`

但 Q/V、transition、OPE 用的 action 却不是“整段 chunk”，而只是 chunk 的第一个低层动作。  
这意味着：

- critic 学到的是 `Q(s, a_first)`，不是论文里的 `Q(s, a_chunk)`；
- PPO 用 `A(s, a_first)` 去更新生成整段 action trajectory 的 diffusion policy；
- OPE gate 评估的也不是 chunk policy 的值。

### 影响

这是当前实现和论文最核心的不一致之一。  
只要任务处于 chunk mode，优势估计、policy improvement、OPE gate 都在优化错误的决策对象。

### 修复建议

- 统一 chunk-level action 表示。
- critic / transition / OPE 的 action 输入应改为实际执行的 chunk：
  - 例如 `[n_action_steps, action_dim]` 展平后输入；
  - 或单独定义 chunk critic head。
- PPO 的 shared advantage 也应对应同一个 chunk 决策，而不是只对应第一个 action。

## P0: OPE gate 里的 AM-Q 不是论文定义的多步模型 rollout

### 论文要求

论文 Eq.(20) 的 OPE gate 依赖 learned transition model 上的 `H` 步 rollout，比较候选策略和当前 behavior policy 的 `AM-Q` 估计值，再决定 accept / reject。

### 当前代码证据

- OPE 主逻辑：
  - `diffusion_policy_3d/rl100_trainer.py:651-723`
- 关键近似写在注释里：
  - `diffusion_policy_3d/rl100_trainer.py:692-705`

具体行为是：

1. `h=0` 时，用真实 `obs_dict` 过一次 `policy.predict_action()`。
2. `h>0` 时，由于“无法从 feature 还原 obs_dict”，直接采样随机归一化动作：
   - `diffusion_policy_3d/rl100_trainer.py:699-705`
3. 然后把随机动作丢给 Q 和 transition model。

### 额外问题

transition model 本身在 chunk mode 下也不是学的 chunk transition，而是：

- 输入只用第一个 action；
- target 是 `t -> t+1` 的 feature shift，而不是 chunk-level `t -> t+n_c`：
  - `diffusion_policy_3d/model/rl/transition_model.py:297-324`

所以 OPE 这里不只是“rollout 近似”，而是 `policy action` 和 `world model dynamics` 两端都没有和论文的 chunk MDP 对齐。

### 影响

OPE gate 现在无法承担论文里“保守单调改进”的角色。  
accept / reject 的结果会高度依赖这套近似，尤其在 `h>0` 以后已经不再是 policy-conditioned rollout。

### 修复建议

- 让 transition model 预测 chunk-level next feature。
- 让 OPE 能在 latent / feature 空间上调用 policy，而不是第 2 步后退化成随机动作。
- 如果做不到 feature-space policy rollout，就不要把当前实现标注为论文 Eq.(20) 的 AM-Q。

## P0: Online RL 阶段没有真正实现论文 Eq.(21)

### 论文要求

论文 Stage 3 是 on-policy fine-tuning：

- 优势来自 `GAE(lambda, gamma; r_t, V_psi)`；
- policy loss 是 online PPO 目标；
- value loss 对应 Eq.(21)；
- 这是离线阶段之后的最后一段 on-policy 提升。

### 当前代码证据

- Online 阶段入口：
  - `diffusion_policy_3d/rl100_trainer.py:1108-1255`
- 代码确实计算了 GAE：
  - `diffusion_policy_3d/rl100_trainer.py:1135-1212`
- 但 policy 更新时并没有使用 `all_advantages`：
  - `diffusion_policy_3d/rl100_trainer.py:1245-1252`

当前实际做法是：

1. 先用在线数据更新 critics；
2. 然后把在线 episode merge 回 dataset；
3. 再直接调用 `offline_rl_optimize()`。

而 `offline_rl_optimize()` 内部重新计算的是：

- `advantages = Q(s, a) - V(s)`
  - `diffusion_policy_3d/rl100_trainer.py:812-820`

这不是 online GAE objective。

### 还有两个实现偏差

- 在线 critic 更新里，Q 的 `next_obs` 直接用 `all_obs_feats[idx+1]` 近似：
  - `diffusion_policy_3d/rl100_trainer.py:1225-1229`
- 这些 `idx` 来自打乱后的全局拼接张量，已经不再保证同一 episode 的真实后继状态。

### 影响

即使把 `run_online_rl` 打开，当前 Stage 3 也不能视为论文里的 online RL。  
更准确地说，它是“先用在线数据改 critic，再回到 offline objective 做一轮重优化”。

### 修复建议

- 单独实现 online PPO 更新函数。
- policy update 必须直接消费在线 rollout 的 old log-prob、GAE advantage、return。
- critic update 要保留真实 episode 边界和真实 next state，不能用乱序 `idx+1` 近似。

## P1: PPO 仍把最后一个 deterministic DDIM step 当成 Gaussian step 参与 likelihood ratio

### 论文要求

论文在 diffusion / RL 推导部分明确指出：  
当某一步方差退化为 0 时，该步是 deterministic mapping，Gaussian log-density 不再成立，因此 likelihood-based objective 只能用于“严格正方差”的步骤。

### 当前代码证据

- 最后一步 `t_prev < 0` 时，代码把 `alpha_t_prev` 设为 1：
  - `diffusion_policy_3d/policy/rl100_policy.py:154-159`
- 然后在 `use_variance_clip=True` 时，会把理论上为 0 的方差裁到 `sigma_min^2`：
  - `diffusion_policy_3d/policy/rl100_policy.py:161-167`
- 采样时最后一步虽然强制用 `mean`，不再加噪声：
  - `diffusion_policy_3d/policy/rl100_policy.py:276-283`
- 但 log-prob 仍继续按 Gaussian 计算：
  - `diffusion_policy_3d/policy/rl100_policy.py:285-286`
- PPO 新旧策略比值也把该步纳入：
  - `diffusion_policy_3d/policy/rl100_policy.py:414-482`

### 问题本质

这里相当于把最后一个 deterministic step“伪装成”一个方差为 `sigma_min^2` 的随机 step。  
这会让 PPO objective 比论文多出一个并不存在的 stochastic term。

### 影响

这不是最致命的问题，但会系统性改变 log-ratio 的数值和梯度来源。  
如果你后面要认真对齐论文公式，这一步必须改。

### 修复建议

- 最后一个 deterministic step 不参与 log-prob / ratio；
- 或者仅对严格正方差步骤累加 PPO surrogate。

## P1: Consistency model 没有接入主评测路径，且 OPE reject 时不会回滚

### 论文要求

论文把 consistency model 作为 one-step deployment policy，并在算法输出里同时保留 `pi_ddim` 和 `pi_cm`。  
离线 / 在线训练里，consistency distillation 也是和 RL 共同推进的。

### 当前代码证据

- consistency model 训练入口存在：
  - `diffusion_policy_3d/rl100_trainer.py:140-157`
  - `diffusion_policy_3d/rl100_trainer.py:834-850`
- 但 eval 和 rollout 路径始终只调用 `policy.predict_action()`：
  - `diffusion_policy_3d/env_runner/metaworld_runner.py:86-96`
  - `diffusion_policy_3d/env_runner/metaworld_runner.py:192-205`
  - `eval_rl100.py:46-71`
  - `train_rl100.py:155-163`
- `ConsistencyModel.predict_action()` 存在，但没有被 env runner 接入：
  - `diffusion_policy_3d/model/rl/consistency_model.py:113-141`

另外，OPE reject 时只回滚了：

- `policy_snapshot`
- `ema_snapshot`

而没有回滚 `consistency_model`：

- snapshot 保存处：`diffusion_policy_3d/rl100_trainer.py:755-757`
- reject 回滚处：`diffusion_policy_3d/rl100_trainer.py:897-902`

### 影响

- 论文“one-step high-frequency deployment”这一条，在当前主流程里还没有真正落地。
- 一旦某次 offline RL update 被 OPE 拒绝，consistency model 仍可能停留在“被拒绝 teacher”的蒸馏结果上，和最终接受的 DDIM policy 失配。

### 修复建议

- 为 consistency model 提供和 `BasePolicy` 对齐的 inference wrapper。
- 在 eval / collect / deploy 中显式支持 `ddim` 和 `cm` 两种策略。
- OPE reject 时同步回滚 consistency model 和其 optimizer state。

## P2: 默认配置并不对应论文完整 recipe，也无法开箱复现

### 当前代码证据

- 默认关闭 online RL：
  - `diffusion_policy_3d/config/rl100.yaml:165-170`
- 默认关闭 ReconVIB：
  - `diffusion_policy_3d/config/rl100.yaml:64-71`
- 默认开启 resume，且路径指向仓库外部绝对路径：
  - `diffusion_policy_3d/config/rl100.yaml:181-183`

### 影响

这会带来两个直接问题：

1. 默认配置不是论文的完整三阶段 recipe。
2. 在一台干净机器上，默认命令大概率会因为外部 checkpoint 路径无效而直接失败。

### 修复建议

- 提供一套“论文对齐配置”：
  - `run_online_rl=true`
  - 明确的 ReconVIB 训练方案
  - `resume=false`
- 把 resume path 移到命令行覆盖，而不是写死在默认 yaml。

## 不是问题的点

以下部分我核对后认为是对齐或基本合理的，不应和上面的硬问题混淆：

- offline RL 时冻结 encoder：
  - `diffusion_policy_3d/rl100_trainer.py:767-770`
  - 这和论文正文里“offline RL 共享并冻结 IL encoder”是一致的。
- `epsilon` prediction：
  - `diffusion_policy_3d/config/rl100.yaml:50-54`
  - 与论文在 RL post-training 中偏好 `epsilon` parameterization 一致。
- `L_total = L_RL + lambda_CD * L_CD` 的联合优化形式已经实现：
  - `diffusion_policy_3d/rl100_trainer.py:834-850`

## 修复优先级建议

如果你的目标是“先把论文主结论跑通”，建议按下面顺序修：

1. 先修 `chunk-action` 表示统一。
2. 再修 OPE / transition model 的 chunk-level rollout。
3. 单独补齐真正的 online PPO + GAE。
4. 最后再处理 final deterministic step 的 log-prob 和 consistency deployment。

如果你的目标只是“先做一个能稳定提升 Meta-World 的近似版”，那至少也要先修前 3 项；否则即便曲线变好，也很难说是在复现 RL100。
